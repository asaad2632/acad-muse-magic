# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

AcadArchiv (الأرشيف الأكاديمي) — an Arabic-language (RTL) research/archiving tool built for a
single PhD thesis ("الخليج العربي في سنوات الحرب العالمية الثانية 1939-1945"). It's a
tab-based SPA for managing archive documents, bibliography, translations, flashcards, and a
"Supervisor Room" workflow shared between a researcher and supervisor account, backed by
Supabase (Postgres + Auth + Storage) and an LLM chat/analysis feature proxied server-side.

## Commands

Package manager is **bun** (`bun.lock`, `bunfig.toml`) — use `bun run <script>`, not npm/yarn.

- `bun run dev` — start the Vite dev server (TanStack Start, port 3000)
- `bun run build` — production build (nitro preset: netlify, output to `.output/public`)
- `bun run preview` — preview a production build
- `bun run lint` — ESLint over the whole repo
- `bun run format` — Prettier write (100 print width, double quotes, trailing commas — see `.prettierrc`)

There is no test suite in this repo.

`bunfig.toml` enforces a 24h supply-chain guard on new package installs (`minimumReleaseAge`).
Only a couple of `@lovable.dev/*` packages are excluded; adding more excludes requires user
confirmation.

## Architecture

### The app is one file, not the route tree

Despite using TanStack Start's file-based router (`src/routes/`, see `src/routes/README.md`
for the routing conventions — do not create `src/pages/` or Next/Remix-style files), almost
all product logic lives in **`src/App.jsx`** (~5,700 lines), rendered as a single client
component from `src/routes/_authenticated/index.tsx`. It's a tab-based UI driven by a
`navItems` array (search for `const navItems`) — tabs include: home, thesis structure, document
search, flashcards, translator, dictionary, my library, URL import, add source, export,
bibliography, defense simulator, AI assistant, and supervisor room. Each tab's read/write path
is a section within this one component, not a separate file.

When making changes here, prefer surgical edits routed through shared helpers over rewriting a
tab from scratch — this file is intentionally kept as one component per prior project decisions
(see `.lovable/plan.md` for the reasoning behind the current source-schema layer).

Supporting modules imported by `App.jsx`:
- `src/config.js` — AI model list/selection (persisted to localStorage)
- `src/aiClient.js` — `callLLM()` / `analyzeDocumentLLM()`, thin wrappers over the server proxy
- `src/cloudSync.js` — the entire Supabase read/write layer (see below)
- `src/SupervisorRoom.jsx`, `src/NotificationBell.jsx` — the two split-out feature components
- `src/lib/excelImport.js` — Excel import parsing for the library

### Routing

- `src/routes/__root.tsx` is the only root layout/shell (`<html dir="rtl" lang="ar">`, Cairo
  font, `QueryClientProvider`). Every route renders inside its `<Outlet />`.
- `src/routes/_authenticated/route.tsx` gates everything under it on a Supabase session
  (`beforeLoad` redirects to `/auth` if there's no user); `_authenticated/index.tsx` mounts
  `App.jsx` client-side only (SSR is skipped for the authenticated shell).
- `src/routes/auth.tsx` / `reset-password.tsx` are the unauthenticated entry points.
- `src/routes/api/ai-chat.ts` is the only backend route: a POST handler that proxies chat
  completions to Groq, OpenRouter, or the Lovable AI Gateway (Gemini), chosen by model-id
  prefix, with retry/backoff and a Lovable-Gemini fallback on 429/5xx. API keys
  (`GROQ_API_KEY`, `OPENROUTER_API_KEY`, `LOVABLE_API_KEY`) are server-only env vars — never
  call these providers directly from client code, always go through this route.
- `src/routeTree.gen.ts` is generated by the TanStack router plugin — never hand-edit it.

### Supabase layer

`src/integrations/supabase/*.ts` files are all marked "automatically generated — do not edit
directly" and should be regenerated via Supabase tooling rather than hand-patched:
- `client.ts` — browser client (publishable key), used from `App.jsx`/`cloudSync.js`
- `client.server.ts` — service-role admin client; server-only, bypasses RLS, must be dynamically
  imported inside server handlers (not top-level-imported from route/`*.functions.ts` files,
  which ship to the client bundle)
- `auth-attacher.ts` / `auth-middleware.ts` — attach the bearer token to server-function RPCs
  (`auth-attacher` is registered as global `functionMiddleware` in `src/start.ts`)
- `types.ts` — generated DB types

`src/cloudSync.js` is organized in phases (3a/3b/3c/3d/4), each covering a group of tables:
chapters/sources/deleted-base-docs, `library_sources` (+ Storage bucket `thesis-files`),
bibliography/cards/translations/custom_formats/researcher_analysis, the Supervisor Room tables
(`supervisor_questions/notes/decisions/files/meetings/reports`), notifications
(`notification_reads`), and an access-control allowlist (`allowed_users`). Two recurring
patterns to follow when adding new synced data:
1. **Shared-workspace tables** (chapters, sources, library_sources, bibliography, cards, ...):
   dedupe rows by `client_id`, latest `updated_at` wins; upsert on `(user_id, client_id)`; on
   delete, remove any DB row whose `client_id` is no longer present locally.
2. **Owner-scoped tables** (the `supervisor_*` tables, keyed by `created_by`/`uploaded_by`):
   only upsert/delete rows the current user owns, to avoid stomping another user's writes in
   the shared workspace.

Migrations live in `supabase/migrations/`, named by phase (e.g. `..._phase3_notifications_and_
phase4_allowlist.sql`).

### Deployment shims — not real app code

`backend/server.py` (FastAPI) and `frontend/package.json` exist only to satisfy the Emergent
Cloud Build deployment template, which expects a `/app/backend` FastAPI service and a separate
`/app/frontend`. `backend/server.py` just reverse-proxies everything to the real TanStack Start
Node server on port 3000 and exposes `/api/health`; `frontend/package.json` just delegates
`install`/`build`/`start` back to the repo root. Don't add real business logic to either — the
actual app is the single-package TanStack Start app at the repo root. Actual deployment target
is Cloudflare Pages (`vite.config.ts`: `nitro.preset: "cloudflare-pages"`; `bun run build`
outputs `dist/`, including the generated `dist/_worker.js/wrangler.json`). There is no
`netlify.toml` in this repo anymore — Netlify was the deployment target prior to this switch.

## Repo conventions

- This repo is connected to Lovable.dev (see `AGENTS.md`) — avoid rewriting published git
  history (force-push, rebase/amend/squash of pushed commits), since that breaks history on
  Lovable's side.
- UI copy and in-app content is Arabic/RTL; expect Arabic identifiers/comments in data tables
  (e.g. `CHAPTERS_DATA`, `DOCS_FROM_INDEX` in `App.jsx`).
- `tsconfig.json` only type-checks `src/**/*.ts(x)` — `App.jsx` and other `.js`/`.jsx` files are
  untyped (see `src/jsx.d.ts` for the blanket `*.jsx` module declaration used to import them
  from `.tsx` route files).
- `src/components/ui/*` is the shadcn/ui component set (Radix + Tailwind) available via the
  `@/*` path alias; most of `App.jsx`'s own UI is hand-rolled inline styles rather than these
  components.
