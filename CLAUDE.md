# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

AcadArchiv (الأرشيف الأكاديمي) — an Arabic-language (RTL) research/archiving tool built for a
single PhD thesis ("الخليج العربي في سنوات الحرب العالمية الثانية 1939-1945"). It's a
tab-based SPA for managing archive documents, bibliography, translations, flashcards, and a
"Supervisor Room" workflow shared between a researcher and supervisor account, backed by
self-hosted DigitalOcean infrastructure (Managed Postgres + Spaces object storage) and an LLM
chat/analysis feature proxied server-side. There is no Supabase dependency anywhere in this
repo — an earlier version of the app used Supabase (Postgres + Auth + Storage); it has been
fully decommissioned and replaced.

## Commands

Package manager is **bun** (`bun.lock`, `bunfig.toml`) — use `bun run <script>`, not npm/yarn.

- `bun run dev` — start the Vite dev server (TanStack Start, port 3000)
- `bun run build` — production build (nitro preset: node-server, output to `.output/`)
- `bun run start` — run the production build (`node .output/server/index.mjs`)
- `bun run preview` — preview a production build
- `bun run lint` — ESLint over the whole repo
- `bun run format` — Prettier write (100 print width, double quotes, trailing commas — see `.prettierrc`)

There is no test suite in this repo.

`bunfig.toml` enforces a 24h supply-chain guard on new package installs (`minimumReleaseAge`).
Only a couple of `@lovable.dev/*` packages are excluded; adding more excludes requires user
confirmation.

## Architecture

### The app is one file, not the route tree

Despite using TanStack Start's file-based router (`src/routes/`, see `src/routes/README.md`
for the routing conventions — do not create `src/pages/` or Next/Remix-style files), almost
all product logic lives in **`src/App.jsx`** (~5,700 lines), rendered as a single client
component from `src/routes/_authenticated/index.tsx`. It's a tab-based UI driven by a
`navItems` array (search for `const navItems`) — tabs include: home, thesis structure, document
search, flashcards, translator, dictionary, my library, URL import, add source, export,
bibliography, defense simulator, AI assistant, and supervisor room. Each tab's read/write path
is a section within this one component, not a separate file.

When making changes here, prefer surgical edits routed through shared helpers over rewriting a
tab from scratch — this file is intentionally kept as one component per prior project decisions
(see `.lovable/plan.md` for the reasoning behind the current source-schema layer).

Supporting modules imported by `App.jsx`:
- `src/config.js` — AI model list/selection (persisted to localStorage)
- `src/aiClient.js` — `callLLM()` / `analyzeDocumentLLM()`, thin wrappers over the server proxy
- `src/cloudSync.js` — the entire data-sync layer, talking to `/api/data/*` (see below)
- `src/SupervisorRoom.jsx`, `src/NotificationBell.jsx` — the two split-out feature components
- `src/lib/excelImport.js` — Excel import parsing for the library

### Routing

- `src/routes/__root.tsx` is the only root layout/shell (`<html dir="rtl" lang="ar">`, Cairo
  font, `QueryClientProvider`). Every route renders inside its `<Outlet />`.
- `src/routes/_authenticated/route.tsx` gates everything under it on the app's own session
  cookie (`beforeLoad` fetches `/api/session`, redirects to `/auth` if there's no user);
  `_authenticated/index.tsx` mounts `App.jsx` client-side only (SSR is skipped for the
  authenticated shell). See "Auth layer" below.
- `src/routes/auth.tsx` is the unauthenticated login entry point. There is no self-registration
  and no email-based password reset — see `scripts/reset-password.ts`.
- `src/routes/api/*.ts` are backend routes (POST/GET/DELETE/PATCH handlers via
  `server.handlers`): `ai-chat.ts` proxies chat completions to Groq, OpenRouter, or the Lovable
  AI Gateway (Gemini), chosen by model-id prefix, with retry/backoff and a Lovable-Gemini
  fallback on 429/5xx; `gemini-analyze.ts` is a separate Gemini-only analysis endpoint;
  `spaces-storage.ts` proxies DigitalOcean Spaces uploads/downloads/deletes (see
  `src/spacesStorage.js`); `login.ts` / `logout.ts` / `session.ts` are the auth endpoints (see
  "Auth layer" below); `data/*.ts` are the data-access endpoints (see "Data access layer"
  below). API keys (`GROQ_API_KEY`, `OPENROUTER_API_KEY`, `LOVABLE_API_KEY`, `GEMINI_API_KEY`)
  are server-only env vars — never call these providers directly from client code, always go
  through a route.
- `src/routeTree.gen.ts` is generated by the TanStack router plugin — never hand-edit it.

### Auth layer

Login is backed by a local `users` table in DigitalOcean Managed Postgres (`db/schema.sql`) —
fully self-contained, no external auth provider.

- `src/db.server.ts` — server-only `pg.Pool`, lazy-Proxy pattern.
- `src/passwordHash.server.ts` — zero-dependency `node:crypto` scrypt hashing.
- `src/session.server.ts` — signed+encrypted session cookie via TanStack Start's built-in
  `useSession`/`getSession`/`updateSession`/`clearSession` (`@tanstack/react-start/server`,
  h3-backed) — not hand-rolled crypto, this primitive ships with the framework already.
  Session payload is `{sub, role, tokenVersion}`; `token_version` on the `users` row lets an
  admin password reset instantly invalidate all previously-issued cookies for that user.
- `src/routes/api/login.ts` / `logout.ts` / `session.ts` — the three auth endpoints. `session.ts`
  is a probe (`{user}` or `{user: null}`, always 200), used by the router guard and `App.jsx`'s
  header/role display — not a gate itself.
- `scripts/seed-users.ts` — one-time: prompts for email/role/password per account and inserts
  into `users` (id is DB-generated). Safe to re-run (already-seeded emails are skipped).
- `scripts/reset-password.ts` — admin CLI password reset (no email delivery): force-sets a new
  password hash and bumps `token_version`, invalidating existing sessions for that user.

Required env vars: `DATABASE_URL` (DigitalOcean Managed Postgres), `SESSION_SECRET` (≥32-char
random string, e.g. `openssl rand -hex 32`).

### Data access layer

`src/dataAccess.server.ts` holds the query helpers backing every `src/routes/api/data/*.ts`
route (one route file per table or small group; table names are always hardcoded literals at
the call site, never derived from request input). Every route is auth-gated by
`getSessionUser()` (`src/session.server.ts`) — 401 if no session. `src/cloudSync.js` is the
client-side counterpart: every exported function fetches/posts to these routes instead of
calling a database directly, and keeps a stable name/signature so `App.jsx`,
`SupervisorRoom.jsx`, and `NotificationBell.jsx` never talk to `/api/data/*` themselves.

Two recurring access patterns, mirrored between `cloudSync.js` and `dataAccess.server.ts`:
1. **Shared-workspace tables** (`chapters`, `sources`, `library_sources`, `bibliography`,
   `cards`, `translations`, `custom_formats`): fully collaborative — reads are unfiltered (both
   accounts see one merged workspace, client-side dedupe by `client_id`/latest `updated_at`
   wins), writes upsert on `(user_id, client_id)` and delete any DB row whose `client_id` is no
   longer present in the pushed set. The server always stamps `user_id` from the session,
   regardless of anything the client sends — `user_id` here is "who last touched this row," not
   an access-control boundary.
2. **Owner-scoped tables** (the `supervisor_*` tables, keyed by `created_by`/`uploaded_by`):
   same shared read, but a row's authorship is respected — `cloudSync.js` only ever pushes the
   caller's own rows (filtered client-side by `ownerId`) so another account's items are never
   resent/reassigned; the server also only ever upserts/deletes rows scoped to the
   authenticated caller either way.

`chapters` (upsert-only, no delete), `deleted_base_docs` (wipe+insert union set), and
`researcher_analysis` (wipe+insert, keyed by `chapter_id`/`section_id` instead of `client_id`)
are bespoke, not the generic pattern above. `library_sources` also has bespoke per-row
insert/update/delete/bulk-delete endpoints (`/api/data/library`) since the Library UI writes
immediately rather than through a debounced whole-set sync.

`supabase/migrations/*.sql` is historical — the live schema is `db/schema.sql`, run directly
against `DATABASE_URL`.

### Deployment shims — not real app code

`backend/server.py` (FastAPI) and `frontend/package.json` exist only to satisfy the Emergent
Cloud Build deployment template, which expects a `/app/backend` FastAPI service and a separate
`/app/frontend`. `backend/server.py` just reverse-proxies everything to the real TanStack Start
Node server on port 3000 and exposes `/api/health`; `frontend/package.json` just delegates
`install`/`build`/`start` back to the repo root. Don't add real business logic to either — the
actual app is the single-package TanStack Start app at the repo root. Actual deployment target
is a plain Node.js server on DigitalOcean (Droplet/App Platform) — `vite.config.ts`:
`nitro.preset: "node-server"`; `bun run build` outputs `.output/`, and `bun run start` (`node
.output/server/index.mjs`) serves it. There is no `netlify.toml`/`wrangler.toml`-driven deploy
in this repo anymore — Netlify then Cloudflare Pages were the deployment targets prior to this
switch (`wrangler.toml` may still exist with stale Cloudflare `[vars]`; do not rely on it).

## Repo conventions

- This repo is connected to Lovable.dev (see `AGENTS.md`) — avoid rewriting published git
  history (force-push, rebase/amend/squash of pushed commits), since that breaks history on
  Lovable's side.
- UI copy and in-app content is Arabic/RTL; expect Arabic identifiers/comments in data tables
  (e.g. `CHAPTERS_DATA`, `DOCS_FROM_INDEX` in `App.jsx`).
- `tsconfig.json` only type-checks `src/**/*.ts(x)` — `App.jsx` and other `.js`/`.jsx` files are
  untyped (see `src/jsx.d.ts` for the blanket `*.jsx` module declaration used to import them
  from `.tsx` route files).
- `src/components/ui/*` is the shadcn/ui component set (Radix + Tailwind) available via the
  `@/*` path alias; most of `App.jsx`'s own UI is hand-rolled inline styles rather than these
  components.
